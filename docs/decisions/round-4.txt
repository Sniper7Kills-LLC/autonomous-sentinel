STORAGE / CANONICAL FORMAT
================================================================

Q143: You flagged storage cost concern on Q124 (multi-copy ballooning). Two real options:
  (a) **Drop original after transcoding.** Pre-process Lambda transcodes incoming upload to one canonical file (e.g. MP3 96 kbps mono — small, browser-playable, accepted by all 4 transcription backends). Original deleted. ~10x storage savings vs WAV. Lossy — irreversible quality loss.
  (b) **Keep original + canonical.** Original (WAV/whatever) stored once for archival, plus a derivative for browser/transcription. ~1.1-1.5x storage cost vs original-only. Reversible if we ever change canonical format.
  (c) **No canonical.** Use whatever the user uploaded; pre-process Lambda transcodes in-memory at transcribe time, never persists a derivative. Storage = original only; CPU cost paid every transcribe call.
  Recommend (a) — biggest cost win, archival fidelity matters less than bandwidth/storage at scale. EAM audio is 8 kHz voice anyway; 96 kbps MP3 is overkill for the source.
A: C - Derive one standard connonical for web view though. Process as needd for pipelines even if saved as a temporary file in S3 thats fine with a delete when published.
